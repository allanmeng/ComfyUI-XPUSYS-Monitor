# 2026-07-10

## PyTorch 2.13 XPU 新 API 调研尝试
### 背景
- 接 v1.0.4 发布后，PyTorch 2.13 发布了 XPU Device Telemetry 新 API
- 涉及接口：`torch.xpu.memory_usage()` / `utilization()` / `power_draw()` / `clock_rate()` / `temperature()` / `device_memory_used()`
- 底层依赖 `pyzes`（Python Level Zero Sysman 绑定）

### pyzes 兼容性问题
- `pyzes` 硬编码查找 `ze_loader64.dll`，但本机驱动只装了 `ze_loader.dll`（无 64 后缀）
- 安装成功后初始化时获取全局锁不释放，导致 `torch.xpu.memory_usage()` 永久卡死
- 已在 oneapi-src/level-zero 提交 issue

### 尝试过程
1. 用 ctypes 直接调 `zesMemoryGetBandwidth` 替代 — 搞定自定义实现
   - struct 偏移算错一次（忘了 `zes_mem_bandwidth_t` 无 stype 头）
   - 修复后功能正常
2. 双路径设计：`torch.xpu.memory_usage()` 优先，ctypes 兜底
3. 因 pyzes 卡死问题需加超时 + 第一次失败永久禁用标志

### 结果：用户撤回该功能
- 不认可 pyzes 一次性判断的禁用方案
- 也不接受每次 poll 都 3 秒超时
- 最终回退了所有 `gpu_mem_bw_pct` 相关改动
- `git checkout HEAD --` 恢复 7 个文件到 v1.0.4

### 结论
- `pyzes` 兼容性问题解决前，不接入 PyTorch 2.13 XPU telemetry 新 API
- 现有 Level Zero ctypes 路径稳定可靠，不做迁移
